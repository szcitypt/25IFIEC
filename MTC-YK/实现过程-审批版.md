# MTC 实现过程提炼（审批版）

> MnemoTranscode / MTC：**记忆数字化与 AI 关系档案平台**。本文系从设计与代码落地过程中提炼的步骤说明，供课程审批使用；细节以仓库内 `SPEC.md`、`ARCHITECTURE.md`、`README.md` 为准。

---

## 一、阶段划分

### 1. 需求与领域建模

- 明确核心对象：**用户 → 档案（多类型）→ 成员 → 记忆 / 媒体 / 胶囊** 等业务实体。  
- 将「输入（文字 / 多媒体）— AI 整理与检索 — 输出（时间线 / 对话 / 故事书 / 胶囊）」归纳为分层能力，写入 `SPEC.md`。  
- 多渠道（Web、规划中的 QQ 等）在规格中单列为统一路由到对话与记忆服务，避免渠道逻辑散落。

### 2. 技术选型与脚手架

- **前端**：React 18 + TypeScript + Vite + Tailwind，TanStack Query 管请求缓存，Zustand 管轻状态，路由 React Router v6。  
- **后端**：FastAPI + Pydantic v2 + SQLAlchemy 2.0 异步，`alembic` 管理 schema 演进。  
- **基础设施**：PostgreSQL（主存储）、Redis（缓存与 Celery broker）、Qdrant（记忆向量检索）、MinIO（私有媒体桶 + 预签名上传）。  
- **编排**：`infra/docker-compose.yml` 定义全栈与健康检查；`scripts/` 提供一键脚本与 Windows 稳定开发路径。

### 3. 纵向切片实现顺序（典型）

1. **认证与用户域**：JWT、`/api/v1/auth`，为后续所有权与配额打基础。  
2. **档案与成员**：CRUD 与前端页面串联，奠定导航与数据结构。  
3. **记忆与元数据**：情感标签、时间地点等字段与列表/详情交互。  
4. **媒体上传**：后端预签名 URL + MinIO，`frontend` 相册/灯箱与进度反馈。  
5. **向量与语义检索**：写入 Qdrant、检索接口打通「自然语言搜记忆」。  
6. **AI 对话**：LLM Service 注入记忆上下文，前端流式/打字机体验。  
7. **衍生能力**：故事书生成与导出、交互时间线筛选、记忆胶囊（定时解封逻辑）等按需迭代。  
8. **扩展渠道**：`kourichat/` 只读挂载，后端 `/api/v1/kourichat` 与微信场景对接（详见主 README）。

### 4. 工程化与交付

- API 契约：OpenAPI (`/docs`) 与手写 `docs/API.md` 对齐主要模块前缀。  
- 设计一致性：`docs/design-system.md` 与 `frontend` 组件目录中的 UI 基座相呼应。  
- 部署：`frontend` 内置 Nginx 模板与 API 上游变量；文档中区分「全容器」与「本机 Vite + Docker 后端」。

---

## 二、架构要点（与实现对齐）

- **分层**：客户端（Web / 未来 QQ）→ 网关（可选 Nginx）→ FastAPI → 服务层（LLM / 向量 / 媒体 / 渠道等）→ 多后端存储。  
- **异步与任务**：耗时任务可走 Celery，与 Redis 解耦峰值。  
- **安全**：媒体私有桶；敏感配置仅 `.env`；提交包 **`MTC-YK`** 不含真实 `.env`。
